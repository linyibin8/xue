# 观察台（/observe）算力与延迟分析

> 回答用户的问题：「开相机半小时→几百张图→提取题目→结构化→还原 HTML/打印空白卷，这套对算力的要求是不是很高？」
>
> **一句话结论：裸跑高，按本方案做不高。** 瓶颈不是钱（电费每场 < 1 分，云 API 每场几美分到几美元），而是那块被 27B 占满的**共享课堂 GPU 的机会成本**。把「去重 + 后台批处理」两招做掉，几百帧的有效负载就压到分钟级、几乎不挤占上课实时流量。

---

## 1. 硬件与单位成本基线（实测）

| 项 | 值 | 来源 |
|---|---|---|
| 推理卡 | 2× RTX 4090 = 48GB（显存已被 27B + KV 池占满，~2GB 空闲/卡） | dell 实测 |
| 模型 | Qwen3.6-27B-FP8 VLM，SGLang TP=2，ctx 65536，:39000 | 部署 |
| 单次视觉请求延迟 | **~6.4s**（短输出）；本 Demo 实测 5 题页 **7.2s** | 实测/本次 |
| 解码吞吐 | 28–54 tok/s（取 ~40） | 实测 |
| 输入视觉 token | 1800px 图约 **1500–3000 token**（极端竖图到 ~5k）；受 SGLang `max_pixels` 影响 | 估算 |
| 输出 token | 单题结构化 80–150 token；一页 1–5 题 → 200–600 token | 估算 |

单图视觉 prefill 是**算力密集**型（占大头），所以并发上去后单请求会变慢、聚合吞吐**非线性**。这台机现实可并发约 2–4 路，聚合吞吐约 **0.3–0.5 img/s**，不要指望更高。

---

## 2. 「半小时几百张图」要多少算力？

### 不优化（最坏情况）——这就是「看起来吓人」的来源
- 300 张 × 串行 6.4s ≈ **32 分钟**，几乎等于录制时长本身。
- 若像旧版那样 segment + relayout 分两个端点 = 每图 2 次调用 → **64 分钟**。
- 作为「实时逐帧分析」：**不可行**——相机出帧远快于 6.4s/张，且会饿死正在上课的 realtime 请求。

### 关键事实：几百帧里独立题目极少
半小时开相机 ≈ 几百帧，但语义上：一页作业纸通常 5–15 题，学生半小时顶多翻 3–6 页 / 做 2–4 张卷 → **真正独立题约 10–40 道、独立页面约 6–15 个**。其余 90%+ 是同页连拍近重复、手/笔遮挡帧、移动模糊帧、对焦中帧、过暗帧。

### 优化后（本方案）
- 抓拍 ~500 帧 → 模糊/无纸丢 ~300 → 近重复聚类剩 ~30 代表帧 → **真正过 VLM 15–30 张**。
- 每图**只 1 次** VLM 调用（本次已把 segment+relayout 合并为单次 `observe_extract`）。
- 串行也就 **1.5–3 分钟**；走后台低优先级，用户无感。

> **去重是 10–20 倍的杠杆，是整个方案能否成立的关键。**

## 3. 成本对比：钱从来不是约束

| 方案 | 单场（未去重 300 张） | 去重后（~30 张） | 每张 |
|---|---|---|---|
| **自托管 2×4090**（800W，电价 $0.10–0.15/kWh） | 串行 32 分 ≈ 0.42 kWh ≈ **$0.04–0.06** | 3 分 ≈ **$0.005** | ~$0.0002 |
| 云 Gemini 2.x Flash | ~**$0.12** | ~$0.012 | ~$0.0004 |
| 云 GPT-4o/4.1-mini | ~**$0.18** | ~$0.018 | ~$0.0006 |
| 云 Claude Haiku | ~**$1.1** | ~$0.11 | ~$0.0036 |
| 云 Claude Sonnet / GPT-4o 档 | ~**$4** | ~$0.4 | ~$0.013 |

> 自托管边际现金成本近乎为零，真实代价是**占用唯一推理卡**；云 API 一整场也就几美分到几美元。**选型不看钱，看：学生作业的数据隐私 + 是否想绑外部依赖 + 本地 27B 已够用 → 倾向继续自托管，仅在 GPU 被课堂打满又需突发吞吐时，把后台批量溢出到 Gemini Flash 这类便宜云 VLM 做弹性扩容。**

## 4. 优化建议（按性价比排序）

1. **端上预筛去重**（pHash 感知哈希 + Laplacian 模糊门控 + 文档矩形检测 + 特征距离 VNGenerateImageFeaturePrint）——10–20 倍降负载，纯端侧零 GPU 成本。**必做、优先。** 本 Web Demo 已用 JS canvas aHash 做了弱化版（只对近重复连拍帧有效）；iOS 端用 Vision/Neural Engine 会更强。
2. **后台 low-priority 批处理**——复用现成 `run_with_llm_gate`，本 Demo 所有 VLM 调用走 `LLM_PRIORITY_BACKGROUND`，课堂 realtime 永远抢先；定位为「录完批量回放分析」，非实时。工程量小。
3. **专用 OCR 分工**——当前 48GB 塞不下 DocRes/PP-OCRv5 同卡；改走 CPU 侧 PaddleOCR 或端上 `VNRecognizeText` 做文本/分行，VLM 只接管判断与重排，既省输入又提精度。中期。
4. **小模型 triage**——用一个量化小 VL 先判「是不是题目页」，命中才喂 27B，进一步省 prefill。中期。

## 5. 端侧 vs 云端分工

把「扔帧」放到离 GPU 最远的地方（iPhone/iPad Neural Engine），云端 VLM 只干真正需要推理的活：

- **端侧（零云成本零回传）**：模糊门控、文档矩形检测 + 透视校正、近重复丢弃、端上 OCR 做「这页像不像题目」粗门控。
- **云端 27B**：结构化题目抽取、是否作答、口算判分、印刷题干数字版重排。（历史结论：bbox 是按题序的等距网格不准、几何/读图题会编造标准答案 → 服务端强制降级「不确定」，这部分必须留云端模型 + 服务端兜底，端侧 OCR 给不了。）

---

## 6. 本 Demo 落地的取舍（与本分析一致）

| 决定 | 理由 |
|---|---|
| 每图只 1 次 `analyze_images`（合并 segment+relayout） | 每图 2→1 次，直接砍一半算力 |
| 全程 `LLM_PRIORITY_BACKGROUND` + 同 batch_token 串行 | 给课堂 realtime 让路；防单入口独占 27B |
| 前端 aHash 图级去重 + 服务端 fingerprint/simhash 题级去重 | 把几百张压到几十道独立题 |
| 单批硬上限 40 张题目页 | 去重对「各异页面」无收益，真正兜底靠上限 |
| 几何/读图题强制「不确定」、不展示臆造答案 | 模型硬伤，服务端可靠性闸门 |
| 无状态：临时文件用完即删、不写任何用户表 | 独立 Demo 不污染真实数据 |

**定论：对算力要求「裸跑高、按本方案不高」。** 去重 + 后台两招即可把 300 张的有效端到端降到分钟级，电费/调用费均可忽略，GPU 也基本不挤占上课流量。
